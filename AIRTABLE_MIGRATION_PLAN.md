## 5. Inline Notes Problem (Important)

Both Tasks and Outreach store timestamped call notes as **Notion block children** (callout blocks), not as database properties. The app calls `tasks-notes-list` and `outreach-notes-list` to retrieve these as structured objects.

Airtable has no block/page-content concept. Options:

**Option A (Recommended for launch):** Store notes inline as a long text field (`Notes` or `Call Log`) on the parent record. The notes are appended as `[Timestamp] body` lines, matching the current format. Simple to implement ‚Äî no extra linked table needed.

**Option B (Proper relational):** Create a separate `Task Notes` table linked to `Master Action Board` and a `Lead Notes` table linked to `Outreach`. Each row is one note. Allows sorting, querying, and filtering notes independently. More complex to migrate.

The Netlify functions `tasks-notes-list` and `outreach-notes-list` will need to be updated to read from whichever approach you choose. Until then, they can gracefully return an empty array from Airtable.

## 8. Migration Steps

### Phase 1 ‚Äî Prepare (do this before any migration)
1. Create missing Airtable tables: Goals, Financial, Outreach, Notes
2. Add `Notion Page ID` and `Source Database` fields to all target tables
3. Set all `AIRTABLE_TABLE_*` env vars in Netlify (see `.env.example`)
4. Run `/.netlify/functions/airtable-health` to confirm all tables are visible
5. Set `NOTION_OUTREACH_DB_ID` in Netlify env vars

### Phase 2 ‚Äî Audit (confirms before writing)
```bash
cp .env .env.local   # make sure env vars are available
node scripts/audit-notion-airtable-coverage.mjs
```
Review the generated markdown report. Confirm record counts look right.

### Phase 3 ‚Äî Dry Run Migration
```bash
node scripts/migrate-notion-to-airtable.mjs
# Or single DB:
node scripts/migrate-notion-to-airtable.mjs --db CRM
```
No records are written. Shows field mappings and confirms Notion is reachable.

### Phase 4 ‚Äî Live Migration
```bash
node scripts/migrate-notion-to-airtable.mjs --live
```
Creates or updates records in Airtable. Safe to re-run ‚Äî deduplicates by `Notion Page ID`.

### Phase 5 ‚Äî Update Netlify Functions
Convert each Notion-backed function group to use `_airtable.js` instead of `_notion.js`. See `AIRTABLE_SETUP.md` for the converted function list and field mappings.

### Phase 6 ‚Äî Smoke Test
- Open the dashboard
- Check Contacts, Tasks, Opportunities, Outreach, Goals, Financial tabs
- Confirm data loads from Airtable
- Confirm create/update/delete operations work

### Phase 7 ‚Äî Switch Off Notion
Once all functions are converted and smoke tested:
- Remove `NOTION_TOKEN` from Netlify env vars (but keep for Playbook feature)
- Leave `PLAYBOOK_NOTION_TOKEN` if Playbook is still in use

## 9. Rollback Plan

Since Notion records are **never deleted**, rollback is simple:

1. Revert the changed Netlify functions to their previous Notion versions (git revert or git checkout)
2. Re-deploy to Netlify
3. Airtable data stays intact ‚Äî no cleanup needed
4. Notion data stays intact ‚Äî nothing was touched

The only permanent action is **creating records in Airtable**. Those can be deleted manually from Airtable UI if needed.

## 10. Risk Summary

| Risk | Severity | Mitigation |
|------|----------|-----------|
| Missing Airtable tables cause 500 errors | High | Run health check before switching functions; keep fallback Notion functions until confirmed |
| Field names differ between Notion export and current Airtable columns | Medium | Audit script shows mismatches; field maps in `_airtable.js` are the single source of truth ‚Äî update them if columns differ |
| Inline call notes (task/outreach) lost in migration | Medium | Accept as known gap for v1; historical notes remain in Notion; new notes will write to Airtable Notes field |

| `NOTION_OUTREACH_DB_ID` not set | High | Migration script skips OUTREACH if ID is missing; health check flags it |
| Relations (companyIds, taskIds) don't carry over as linked records | Low | Stored as text IDs in migration v1; can be upgraded to linked records after all tables are populated |
